# 05. 실제로 어디에 쓰이는가

지식 증류가 실무에서 많이 쓰이는 가장 큰 이유는 배포 제약 때문이다. 연구용 최고 성능 모델은 대개 크고 느리다. 서버 비용이 많이 들고, 지연 시간이 길며, 모바일이나 엣지 장치에서는 아예 돌리기 어렵다. 지식 증류는 이런 제약 속에서도 가능한 한 높은 성능을 유지하려는 도구다.

이를 대형 트럭과 골목길에 들어가는 작은 차량의 차이로 생각하면 쉽다. 짐을 가장 많이 싣는 트럭이 항상 모든 길에 맞는 것은 아니다. AI도 비슷하다. 가장 큰 모델이 항상 실제 서비스에 맞는 것은 아니기 때문에, 큰 모델의 능력을 더 작은 모델에 옮기는 과정이 필요하다.

## 핵심 설명
실전 환경에서는 정확도만 중요한 것이 아니다. 응답 속도, 메모리 사용량, 전력 소비, 서버 비용, 디바이스 제약이 함께 중요하다. 지식 증류는 작은 student를 학습시켜 이 제약들을 완화하면서도 teacher의 장점을 일부 유지하려 한다.

대표적인 사용 맥락은 크게 네 가지로 나눌 수 있다. 첫째, 모바일과 엣지 배포다. 메모리와 전력이 제한된 환경에서는 teacher를 그대로 쓰기 어렵다. 둘째, 대규모 서버 추론이다. 요청 수가 많은 서비스에서는 모델이 조금만 가벼워져도 비용 절감 효과가 크다. 셋째, 실시간 응답이 필요한 환경이다. 음성 처리, 실시간 필터링, 현장 장비 분석처럼 지연 시간이 짧아야 하는 경우 student가 더 유리하다. 넷째, 연구 모델의 제품화다. 논문용 대형 모델을 실제 서비스용 모델로 바꾸는 과정에서 지식 증류가 자주 등장한다.

컴퓨터 비전에서는 큰 CNN이나 Vision Transformer를 더 작은 student로 압축해 카메라 장치나 현장 장비에서 돌리기 쉽게 만드는 사례가 많다. 음성 분야에서도 큰 음향 모델의 판단을 작은 모델로 옮겨 실시간 처리 환경에 맞추는 경우가 있다. NLP에서는 DistilBERT, TinyBERT, MiniLM이 대표적이다. 이 모델들은 언어 이해 능력을 가능한 한 유지하면서 속도와 메모리 효율을 높이는 방향을 보여 주었다.

중요한 점은 지식 증류가 온디바이스만을 위한 기술이 아니라는 사실이다. 서버 환경에서도 추론 비용은 매우 중요하다. 하루에 수백만 건의 요청을 처리하는 서비스에서는 모델이 조금만 가벼워져도 비용 차이가 크게 벌어진다. 그래서 지식 증류는 클라우드 서비스 운영 관점에서도 중요한 최적화 기법이다.

## 배포 환경별 판단 기준

| 상황 | 왜 지식 증류가 필요한가 | 대표 효과 | 예시 |
| --- | --- | --- | --- |
| 모바일, 엣지 | 메모리와 전력이 제한됨 | 모델 크기 감소, 추론 속도 개선 | 스마트폰 분류 모델, 카메라 장치 |
| 대규모 서버 추론 | 요청 수가 많아 비용이 큼 | GPU, CPU 비용 절감 | 검색 랭킹, 문장 분류 |
| 실시간 응답 | 지연 시간이 짧아야 함 | latency 개선 | 음성 인식, 실시간 필터링 |
| 연구 모델 제품화 | 연구용 모델이 너무 무거움 | 배포 가능한 student 확보 | BERT를 DistilBERT나 TinyBERT로 전환 |

이 표를 보면 지식 증류는 단순히 성능을 조금 줄이고 속도를 얻는 타협이 아니다. 어떤 환경에서는 아예 배포 가능 여부를 결정하는 요소가 된다.

## 언제 지식 증류를 우선 고려할까
- teacher 성능은 충분하지만 비용이 너무 높을 때
- 실시간성이나 온디바이스 제약 때문에 큰 모델을 그대로 쓸 수 없을 때
- pruning이나 quantization만으로는 정확도 유지가 어려울 때
- 여러 서비스에 같은 판단 기준을 더 작은 student들로 복제하고 싶을 때

반대로 배포 제약이 거의 없고 큰 모델을 그대로 써도 되는 상황이라면, 지식 증류의 우선순위는 낮아질 수 있다. 결국 핵심은 성능 문제가 아니라, 성능과 비용의 비율을 어떻게 설계할 것인가에 있다.

## 심화 박스
산업계에서 DistilBERT, TinyBERT, MiniLM이 널리 언급되는 이유는 단순히 작아서가 아니다. 이 모델들은 지식 증류가 실제 제품 수준의 trade-off를 만들 수 있다는 점을 보여 준다. 즉 지식 증류는 논문 속 테크닉이 아니라, 연구 모델을 서비스 모델로 바꾸는 과정에서 자주 등장하는 중간 다리다.

LLM 시대에도 이 논리는 그대로 이어진다. 거대한 teacher가 만든 응답 품질을 더 작은 모델이 최대한 흉내 내게 만들면, 배포 가능한 제품형 모델을 더 쉽게 만들 수 있기 때문이다.

## 자주 생기는 오해
- 지식 증류는 모바일에서만 필요한 기술이 아니다. 서버 비용 절감에도 매우 중요하다.
- 작은 모델을 만든다고 해서 항상 정확도가 크게 떨어지는 것은 아니다. 잘 설계된 student는 예상보다 높은 성능을 유지할 수 있다.
- 지식 증류는 속도를 위한 꼼수라기보다, 제품 수준에서 성능과 비용의 균형을 설계하는 방법이다.
